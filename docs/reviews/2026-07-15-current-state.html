Current state review

A code, security, visual and playability review of Tailgate: After Hours, followed by the fixes completed in the first v1.1 phase.

Verdict

Technically sound, mechanically promising, visually one pass short of release quality. The deterministic simulation and report system are convincing. The in-world presentation still looks like a greybox, and the production HUD explains the engine more clearly than it guides the player.

The best screen is the Engagement Report. It feels specific to Tailgate and gives the run a satisfying security-consultancy payoff. The weakest surface is the moment-to-moment 3D view, where rooms, staff and objectives are harder to read than the underlying systems deserve.

Security review

No Critical or High application flaw was found. This is a static, single-player game with no backend, account system, analytics or user-generated content, which keeps the exposed attack surface small.

Medium - Browser hardening - vercel.ts:20

What was wrong: the live deployment supplied HSTS but no project-level Content Security Policy, anti-framing rule, MIME sniffing protection, referrer policy or device-permission restrictions.

Why it matters: the game has little hostile data today, but missing headers leave the browser with permissive defaults. A later content feature or compromised asset would have fewer limits around it.

Fix: Phase 1 adds a same-origin CSP with only the local data: and blob: allowances required by bundled fonts and GLTF textures, frame-ancestors 'none', X-Frame-Options: DENY, nosniff, no-referrer, and disables camera, microphone and geolocation. The protected preview was checked with authenticated Vercel tooling and a clean browser console.

Medium - CI supply chain - .github/workflows/ci.yml:16,35,65,80

What was wrong: GitHub Actions used mutable major-version tags, the workflow token had no explicit permission boundary, and gitleaks was downloaded without a checksum.

Why it matters: a compromised action tag or release download could execute inside CI. That creates a path to repository data and any secrets later added to the workflow.

Fix: Phase 1 sets contents: read, pins every JavaScript action to a full commit SHA on the Node 24 runtime, and runs the official Gitleaks container by immutable image digest. The scanner no longer depends on an unchecked runtime download.

Positive security observations

  • DOM output uses textContent. No innerHTML or dynamic code execution was found.
  • The repository secret pass was clean. Environment files are ignored and no environment file appears in Git history.
  • The lockfile is committed. npm audit reports zero vulnerabilities after Phase 1.
  • Vite 7.3.6 and Vitest 4.1.10 sit above the affected ranges in the checked advisories.
Dependency note: the latest @vercel/config release introduced three High advisories through its pinned routing package during implementation. It was rejected. Version 0.0.32 supplies the required typed API without those dependencies, and the final audit is clean.

Phase 1 regression coverage

  • Two consecutive engagements produce independent tick-zero input logs.
  • Corrupt localStorage cannot break the kiosk.
  • Concurrent character requests share one GLB parse, retry failed loads, and return independent skinned rigs.
  • Level disposal is unique, idempotent and never owns materials shared with a replacement.
  • Beam attributes survive repeated updates without normal generation.
  • The extracted HUD preserves Phase 6 output exactly.

Visual review

Already working

  • The kiosk and Engagement Report carry the access-control fiction clearly.
  • Amber and alarm red have disciplined roles.
  • Guard beams make the detection model visible in real space.
  • High contrast and reduced motion are present from the first release.

Holding it back

  • Floors, walls and furniture still read as greybox geometry.
  • The operator and cleaners share the same body and silhouette.
  • Objective cylinders look like development markers.
  • Rooms lack enough props, signage and material contrast to read at a glance.

Phase 1 deliberately leaves these surfaces unchanged. They belong to the planned visual pass, where they can be judged together rather than arriving as scattered styling edits.

Playability review

  1. The HUD needs to serve the player. The production view currently lists FPS, simulation time, device, every door and internal state. Phase 2 should keep those values in development builds and replace them with a compact field readout.
  2. Interaction needs an affordance before input. The server rack and photo targets do not show the required E or A hold until progress has already started.
  3. Doors need names and readable states. The data model is clear, but the in-world panels do not yet explain access or timing well enough.
  4. The post-plant checkpoint can become a loop. A guard near the rack can detain the player repeatedly after restart. The safe checkpoint and guard reset belong in Phase 2.
  5. Run length needs real controller evidence. Automated routes prove completion and determinism, but they cannot settle feel. Craig should judge the 5 to 8 minute target with the pad before patrol or movement values change.
